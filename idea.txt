# Phase 1: Adversarial Technical Audit

- **CKKS noise growth and latency:** The CKKS scheme is leveled (approximate) homomorphic encryption, meaning each ciphertext carries inherent noise that grows with every homomorphic multiplication.  Each multiply and rescale adds error, and *bootstrapping* (resetting the modulus chain) does **not** remove noise in CKKS – it merely restores “levels” at the cost of additional error【44†L142-L150】.  In practice this means even moderate-depth computation (like fusing embeddings, graph convolutions, and an ensemble) will quickly exhaust precision without extremely large parameters.  Moreover, CKKS bootstrapping itself is very expensive – on the order of tens to hundreds of milliseconds per operation.  For example, public benchmarks show single CKKS multiplications taking ~5–10 ms and rotations ~30–50 ms (in SEAL/OpenFHE on a single core)【12†L681-L689】.  A single bootstrap often costs similar time to those primitives【44†L150-L153】.  Thus an encrypted inference that involves many multiplies, rotates, and possibly bootstraps (as a multi-modal ensemble would) is far from “real-time”.  In state-of-the-art FHE demos, even a simple encrypted logistic regression on a few-thousand-dimensional input can take hundreds of milliseconds to seconds per inference on CPU【14†L224-L232】【12†L681-L689】.  With thousands of features (the 3,924-dimensional data here) and complex fusion, meeting a 100 ms latency is essentially impossible without radical breakthroughs.  Key bottlenecks are the heavy cost of rotations (for slot manipulation), ciphertext size (~1,000× plaintext size【14†L178-L183】), and bootstrapping overhead.  In short, the **Principal Cryptographer** would judge “real-time” FHE inference here to be a pipe dream: only an extremely shallow model or very short depth could approach 100 ms, and that requires sacrificing most of the intended computation or relying on specialized hardware/GPU for massive parallelism【12†L681-L689】【44†L150-L153】.

- **Graph topology and APK embedding mapping:** The system must attach APK-derived signals (e.g. a device’s malware embedding) to nodes in the transaction graph.  Naively exposing a device-level identifier (IMEI, AndroidID, etc.) or even a hashed version risks leaking personal data.  To avoid this, banks would need to **privately link** device signatures to account nodes without revealing identities.  One standard technique is *private set intersection (PSI)*: each bank hashes its device IDs and jointly computes the intersection, learning only the common devices【34†L49-L53】【34†L103-L107】.  In practice, a PSI-like protocol (possibly with authorization checks) could let banks detect if a malicious APK is present across accounts without ever sharing the raw IDs.  Alternatively, device signatures could be pseudonymized or pre-encrypted in a secure enclave: each bank tags accounts with an encrypted device token before FHE processing.  In any case, **Graph ML architects** would caution that true cross-bank linking likely requires a secure multi-party step (PSI or MPC) *prior* to the homomorphic inference.  Without it, mapping an APK to a graph node would either trivially expose identity (if done in plaintext) or be infeasible in CKKS (which cannot easily compare or match IDs).  The embedding fusion is also delicate: concatenating a pre-computed APK embedding to node features must ensure that the embedding cannot be inverted to reveal device info.  In sum, one must design a privacy-preserving join (e.g. PSI or trusted preprocessing) to associate APK evidence with graph nodes without leaking PII before encryption【34†L49-L53】.

- **Data leakage and inference attacks:** FHE protects data *during* computation, but it does not inherently hide the *outputs* or prevent query-based attacks.  If an attacker (fraudster) can repeatedly query the encrypted inference (and obtain decrypted scores), they could perform *model extraction* or *membership inference* much like in plaintext ML.  Recent work (SEEK) shows that even “secure” inference protocols using HE/MPC can be broken: an adversary can craft inputs to recover model parameters with only a few dozen queries【22†L53-L62】.  Likewise, **Graph Neural Networks are vulnerable to membership-inference attacks**: attackers can often tell if a given node was in the training graph by analyzing outputs, and the rich structural information in a GNN only makes this easier【37†L49-L58】.  In FHEGuard’s case, a hostile bank or insider with query access could try to deduce if a particular account (or device) was present in the training data, or even recover weights of the XGBoost model by systematic probing.  The encryption does not add noise to the final result beyond approximation error, so the adversary effectively gets the same response they would in plaintext.  In short, FHEGuard must assume the same threats as any ML service: **encrypted inference is not a magic bullet against inversion or extraction**. Without rate-limiting, noise or access controls, a determined fraudster might reverse-engineer parts of the model or infer sensitive links from repeated use【22†L59-L64】【37†L49-L58】.

# Phase 2: Enterprise Viability & Compliance (CISO’s View)

- **Regulatory compliance:** Encrypting data-in-use is a strong technical safeguard, but it does *not* automatically remove banks’ legal obligations.  Under GDPR (UK or EU), homomorphic encryption is treated as pseudonymization, **not full anonymization**【24†L276-L284】.  All the FHEGuard operations still count as “processing personal data” requiring a lawful basis.  In fact, banks will almost always justify AML/CTF analytics under **legal obligation or public interest** (e.g. Article 6(1)(c)/(e) GDPR)【26†L95-L104】.  Likewise, UK AML regs prohibit “tipping off” suspects.  Using FHE does not override that: if an account is flagged as part of a mule ring, the bank still must *not* alert the customer before filing a Suspicious Activity Report.  The difference is mainly technical: FHEGuard could allow sharing encrypted risk scores between banks and regulators *without exposing raw transactions*.  But it cannot replace legal reporting obligations.  In GDPR terms, the data processed by FHEGuard would still be covered by the GDPR (unless it can be proven that no key exists – unlikely).  That means banks need proper consent/notice or rely on AML compatibility exemptions.  They should ensure privacy notices cover this analytics (perhaps disclosing that data may be processed and shared in encrypted form for fraud prevention, per best practice【26†L123-L132】).  On balance, FHEGuard may **reduce breach risk** (a regulator with only ciphertext can’t see identities), but it does *not* absolve the banks of their AML or data-sharing liabilities under laws like the Bank Secrecy Act or GDPR.  It is essentially a “supplementary measure” – helpful for data minimization, but not a free pass.

- **Integration into banking infrastructure:** In practical terms, FHEGuard would sit on the analytics/fraud-monitoring layer, not replacing core banking ledgers.  It is akin to a sophisticated AML/fraud engine that banks or a consortium run.  Conceivably, banks would feed the system encrypted transaction summaries and device signals; the FHEGuard service (maybe operated by a regulator or neutral host) would return encrypted threat scores.  This is conceptually similar to existing collaborative networks (e.g. SWIFT’s sanctions screening, FinCEN’s suspicious transaction database).  However, the real-time requirement (sub-100 ms) is so stringent that it likely precludes on-the-fly scoring for each payment.  More realistically, FHEGuard would be used as a *background analytic* or as part of a clearinghouse: for example, it could flag suspicious cross-bank networks overnight or in-batch, with banks then acting on high-risk cases.  If insisted on real-time, one might try to hook it into payment rails (e.g. SWIFT FIN messages or instant-payment systems like FedNow), but even existing real-time fraud-check systems (e.g. rules/AI engines) typically aim for tens to hundreds of milliseconds without FHE.  In practice, a Global Bank CISO would view FHEGuard as an advanced **fraud-scoring API**: a back-end service that augments (not replaces) the bank’s standard AML engine.  It would be integrated via secure links (APIs or enclave machines) to the transaction monitoring tier, rather than in the core account database.  Any slower processing (FHE homomorphic ops) would likely force it into a nearline or settlement-stage workflow. 

# Phase 3: Hackathon MVP Reality Check

- **Impossible in 2–3 days:** Many components must be *simulated or mocked*.  In practice, building even a small GNN on a 3,924-feature, multi-million-edge graph is nontrivial.  Training a relational GNN (R-GCN) or GraphSAGE takes hours or days on GPUs, and ingesting real transaction data is infeasible in a weekend.  Likewise, a full APK analysis pipeline (static/dynamic analysis to text embedding) is a major project; at best one would stub this with random or precomputed “APK embeddings”.  The CKKS homomorphic part is especially heavy: configuring SEAL/TenSEAL with large parameters, implementing the ensemble under encryption (including bootstrapping!), and getting any output is extremely time-consuming.  Even simple HE inference examples (e.g. logistic regression with TenSEAL) take careful setup.  In a hackathon, the team would almost certainly *skip actual FHE* for most of the workflow.  They might either remove bootstrapping entirely (thus only doing a shallow encrypted computation) or simply “fake” the encryption by running plaintext math and labelling it as encrypted for demo purposes.  In short, expect to replace the real graph and FHE engines with stand-ins: e.g. use a toy graph with a few nodes, a dummy neural network or even a static embedding, and perhaps just demonstrate one encrypted dot-product.

- **Golden-path proof-of-concept:** The MVP should focus on end-to-end data flow rather than full fidelity.  A sensible demo could be: 
   1. **Prep:** Pre-train or hard-code a tiny model (e.g. a 2-node GraphSAGE or even just a weighted sum of features) and an XGBoost tree or logistic regressor on toy data.  
   2. **Simulate APK signal:** For one “malicious APK”, manually create a small 5–10 dimensional embedding vector.  
   3. **Encrypted inference:** Using TenSEAL (or any CKKS library), have a pre-loaded key/params and encrypt a short feature vector (e.g. a 5-D combined input of graph features + APK embedding).  Perform one or two homomorphic operations (like weighted sum or a simple polynomial) to compute a score, then decrypt to show it matches the plaintext version.  
   4. **Visualization:** Emphasize that the result was computed on ciphertext (perhaps show the raw ciphertext in the console for effect).  
   
   Crucially, skip anything that takes seconds: do *not* attempt a real 3-layer GNN under HE, nor a full tree ensemble.  Instead, even one homomorphic multiplication plus addition is enough to “prove” FHE works.  The rest of the pipeline (APK analysis, node lookup) can be trivial or static.  For example, allocate a small synthetic graph and directly attach the APK embedding to one node’s feature.  The goal is to demonstrate *some* encrypted score coming out correctly, not end-to-end scale.  Any cross-bank “secure sharing” can just be mentioned conceptually or shown as a mocked handshake.  In sum, the POC should spend time on a single well-understood encrypted inference (with maybe pre-encrypted model weights) and gloss over or fake the rest. 

# Phase 4: VC Verdict & Recommendations

- **Defensibility (1–10):** I would rate FHEGuard around *4 or 5/10*.  It addresses a genuine problem (cross-bank mule networks) and leverages cutting-edge tech, but the technical moat is weak.  FHE itself is not proprietary – open libraries like Microsoft SEAL or OpenFHE already exist.  A well-funded competitor could instead use Secure MPC or hardware enclaves (SGX) to achieve similar privacy.  In fact, secure enclaves can run unmodified code on encrypted memory *faster and cheaper* than non-interactive HE【41†L299-L305】【41†L414-L422】.  Multi-party computation (MPC) frameworks are also maturing for collaborative analytics.  Unless FHEGuard has novel algorithms or data (which isn’t indicated), the barrier is not huge: the real challenges are engineering and gaining bank cooperation.  On the market side, anti-fraud is large, but “cross-bank mule detection” is a relatively niche segment.  The **Why Now?** factor is arguable: interest in privacy tech is high circa 2026, but adoption hurdles remain.  Overall, FHEGuard’s idea is intriguing, but I’m skeptical it’s a defensible, sustainable business as stated. 

- **Recommendations:** To strengthen the design before full development, consider these changes:
  1. **Hybrid Privacy Engine:** Offload heavy ML to trusted hardware or partitioned compute.  For example, perform the GNN embedding and tree scoring inside secure enclaves (SGX/AMD SEV) on an auditor’s server, rather than fully in FHE.  This avoids deep homomorphic circuits and meets latency better【41†L299-L305】【41†L414-L422】.  Use FHE only for final aggregation or when MP guarantees are needed.  In practice, banks could send encrypted data to a TEE-based service, which computes the score without ever exposing raw data, achieving much lower latency than pure HE.  
  2. **Model Simplification:** Reduce depth and nonlinearity to avoid costly bootstraps.  For instance, replace complex GNN + XGBoost with a single-layer graph embedding plus a linear or low-degree polynomial classifier.  Where possible, pre-compute node embeddings in plaintext within each bank, then homomorphically sum or weight them.  Using linear models (or carefully tailored polynomials) can often be done *without* bootstrapping, dramatically cutting compute.【41†L399-L407】  
  3. **Privacy Hardening:** Add defenses against inversion attacks.  For example, limit query rates, add differential privacy noise to outputs, or discretize scores to prevent exact extraction.  Since GNNs leak membership via graph structure【37†L49-L58】, consider dropping or coarsening edges in the encrypted model, or training with defense methods (per that paper) to blunt membership inference.  Similarly, enforce that the inference service returns only encrypted scores and only to credentialed parties, to curb malicious probing.  
  4. **Secure Data Linking:** Formalize the device-to-account join via cryptographic PSI/MPC.  Instead of hand-waving “embedding fusion,” build a small PSI protocol so that multiple banks can find shared device IDs *without sharing other data*.  This might use existing PSI libraries or protocols, and would be far more robust (and defensible) than an ad-hoc hash.  Ensuring the cross-bank link step is provably private will make the whole design stronger.  

These changes focus on realistic performance and security: by leveraging enclaves or MPC for heavy lifting, trimming model complexity, and hardening against known attacks, FHEGuard would become more viable both technically and in the eyes of regulators.
